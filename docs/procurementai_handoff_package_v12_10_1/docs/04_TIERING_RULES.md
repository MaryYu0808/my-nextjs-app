# Tiering Rules (Phase 1)

Phase 1 includes **Tier 1 + Tier 2** only. Tier 3 is excluded to keep the directory signal high and procurement-specific.

## Tier 1 — Procurement-native (strongest relevance)
**Definition:** The core user and core value proposition are procurement-led. Product positioning explicitly targets procurement workflows.

**Examples (non-exhaustive):**
Coupa, SAP Ariba, GEP, Zycus, Ivalua, Zip, Sirion, EcoVadis, Fairmarkit, Oracle Procurement Cloud, etc.
Also includes procurement-focused AI specialists (contract, sourcing, risk, P2P/AP automation, intake/orchestration).

**Typical evidence:**
- Product pages explicitly mention procurement / sourcing / S2P / P2P / CLM / supplier risk / AP automation
- Buyer persona: CPO, Procurement Ops, Category Manager, AP
- Core modules map directly to one of the 8 workflow categories

## Tier 2 — Procurement-first Adjacent Intelligence (strong relevance, adjacent)
**Definition:** Not a procurement system core, but the primary value proposition serves procurement decision-making and supplier management.
Procurement is a primary (often first) use case.

**Examples (categories):**
- Supplier risk / ESG / third-party risk & sanctions screening (procurement-led)
- Cost intelligence / market intelligence (procurement-led, clearly stated)
- Supplier intelligence for onboarding/monitoring used primarily by procurement

**Typical evidence:**
- Procurement is explicitly named as a target buyer/use case on product pages
- Procurement use cases are prominent in customer stories and GTM messaging
- Maps clearly into the 8 workflow categories (esp. risk/intelligence)

## Tier 3 — Procurement-enabled Horizontal (excluded in Phase 1)
**Definition:** General-purpose platforms (data, BI, automation, warehouses, etc.) that procurement teams can use but are not “built for procurement”.

**Examples:**
Make, Snowflake, Databricks, generic BI/automation/data infra.

**Handling:**
Excluded from Phase 1. Can be added later as a Secondary Layer in Phase 2/3.

## Hard inclusion rule (Phase 1)
If a company satisfies **any 2** of the below, it is generally considered “strongly relevant” (Tier 1 or Tier 2). Otherwise, place it in Secondary Layer (excluded for Phase 1):

1) Product positioning (not blog) explicitly includes: procurement / sourcing / S2P / P2P / supplier risk / CLM / AP automation
2) Product maps clearly into **one** of the 8 workflow categories (single primary category)
3) Customer cases clearly target procurement roles (CPO/Procurement Ops/Category Manager/AP)
4) Core modules are procurement workflow primitives (RFx, CLM, supplier onboarding/risk, invoice/AP, intake/orchestration, negotiation support, spend analytics)
